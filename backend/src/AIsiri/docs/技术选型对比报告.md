# 多智能体框架技术选型对比报告

## 一、候选框架

| 维度 | AutoGen (Microsoft) | CrewAI | LangGraph.js (LangChain) |
|------|---------------------|--------|--------------------------|
| 语言支持 | Python only | Python only | JavaScript / TypeScript |
| 核心范式 | 对话式多智能体 | 角色扮演式多智能体 | 图状态机 |
| 智能体定义 | ConversableAgent | Agent + Task + Crew | StateGraph + Node |
| 协同模式 | 对话链 | 顺序/并行任务流 | 条件路由 + Supervisor |
| 状态管理 | 对话历史 | 共享上下文 | 结构化 State + Checkpoint |
| 持久化 | 无内置 | 无内置 | MemorySaver / PostgresSaver |
| npm 生态 | 不兼容 | 不兼容 | 原生支持 |
| 与 Express 集成 | 需 Python 微服务桥接 | 需 Python 微服务桥接 | 直接 require 使用 |

## 二、选型决策

**最终选择：LangGraph.js v1.2.2**

### 决策依据

1. **技术栈一致性**：本系统后端基于 Node.js/Express，前端基于 Vue 3，LangGraph.js 是唯一原生支持 Node.js 的成熟多智能体框架，无需引入 Python 运行时或微服务桥接层。

2. **架构匹配度**：LangGraph 的 Supervisor 模式天然对应本系统"中央路由与协调智能体"的设计；Graph 中的 Node 对应各专项智能体；条件路由实现动态意图分发。

3. **状态管理**：LangGraph 提供结构化 State（Annotation）与自动 Checkpoint，可直接实现"用户记忆智能体"的长短期记忆需求。

4. **开发效率**：`@langchain/core` 已存在于项目依赖中，仅需追加 `@langchain/langgraph`，增量改动最小。

5. **时间约束**：距论文截止 2.5 个月，AutoGen/CrewAI 方案需要重写后端或搭建微服务架构，风险过高。

### 与开题报告的衔接说明

开题报告中提出使用 AutoGen 框架。经深入技术调研与原型验证，AutoGen 仅支持 Python 生态，与本系统已有的 Node.js/Express 后端存在技术栈不兼容问题。综合考虑系统稳定性与开发效率，最终选择 LangChain 团队推出的 LangGraph.js 作为多智能体协同编排引擎。LangGraph.js 提供图状态机、条件路由、状态持久化和 Supervisor 模式，完整覆盖本文所需的多智能体协同架构需求。

## 三、LangGraph.js 核心概念与本系统映射

| LangGraph 概念 | 本系统映射 |
|---------------|----------|
| StateGraph | 多智能体协同调度图 |
| Annotation (State) | 智能体间共享的通信协议 |
| Node | 各专项智能体（任务/调度/情感/工具） |
| Conditional Edge | 中央路由的意图分发逻辑 |
| Supervisor Pattern | 中央路由与协调智能体 |
| MemorySaver | 用户记忆智能体的状态持久化 |
| Checkpoint | 对话上下文保存与恢复 |
