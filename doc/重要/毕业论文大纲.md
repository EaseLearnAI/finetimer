## 第四章 系统关键模块实现（约 5,000 字，**论文核心章节**）

### 4.1 LangGraph 状态图构建（约 600 字）

**写作要点**：

展示 `graph.js` 的核心代码，解释图的构建过程：

```javascript
const graph = new StateGraph(AgentState);
graph.addNode('loadMemory', loadUserMemory);
graph.addNode('router', routerAgent);
graph.addNode('taskAgent', taskAgent);
graph.addNode('scheduleAgent', scheduleAgent);
graph.addNode('emotionAgent', emotionAgent);
graph.addNode('saveMemory', saveUserMemory);

graph.addEdge(START, 'loadMemory');
graph.addEdge('loadMemory', 'router');
graph.addConditionalEdges('router', routeDecision, {
  taskAgent: 'taskAgent',
  scheduleAgent: 'scheduleAgent',
  emotionAgent: 'emotionAgent',
});
graph.addEdge('taskAgent', 'emotionAgent');
graph.addEdge('scheduleAgent', 'emotionAgent');
graph.addEdge('emotionAgent', 'saveMemory');
graph.addEdge('saveMemory', END);
```

说明关键设计点：
- `addConditionalEdges` 实现动态路由，`routeDecision` 返回目标数组支持多意图并行
- taskAgent / scheduleAgent 执行完毕后汇入 emotionAgent，保证结果整合
- 编译后缓存（`_compiledGraph` 单例），避免重复构建开销

**引用来源**：`backend/src/AIsiri/agents/graph.js`

### 4.2 中央路由智能体实现（约 800 字）

**写作要点**：

1. **核心创新：单次 LLM 三合一路由**

### 4.3 任务管理智能体实现（约 600 字）

**写作要点**：

TaskAgent 采用**纯规则引擎**（不调用 LLM），从 Router 传递的 `extractedEntities` 中提取信息：

1. **时间块解析**（`parseTimeBlock`）：
   - "早上/早晨/清晨" → morning (06:00-09:00)
   - "上午" → forenoon (09:00-12:00)
   - "下午" → afternoon (14:00-18:00)
   - "晚上/晚间" → evening (19:00-22:00)

2. **具体时间解析**（`parseSpecificTime`）：正则匹配 `(\d{1,2})[点时:：](\d{0,2})`，处理 AM/PM 歧义

3. **日期解析**（`parseDate`）：今天/明天/后天的相对日期计算 + ISO 格式绝对日期支持

4. **任务标题提取**：当 LLM 未提取到 tasks 实体时，使用输入文本去除动词前缀后作为标题

5. **持久化**：通过 `Task.create()` 写入 MongoDB

规则引擎的优势：延迟极低（< 5ms）、结果确定性高、便于测试

### 4.4 日程调度智能体实现（约 600 字）

**写作要点**：

ScheduleAgent 采用"LLM 生成指令 + 规则引擎执行"的混合架构：

1. **数据准备**：从 MongoDB 加载用户已有任务列表、任务集信息
2. **LLM 推理**：将现有任务 + 用户请求发送给 qwen-plus（temp=0.3），要求返回操作指令 JSON 数组
3. **指令类型**：
   - `update`：修改时间/优先级/象限
   - `reschedule`：移至新时间块/日期
   - `create`：新建建议性任务
   - `delete`：删除重复/过期任务
   - `reorder`：调整顺序
4. **执行引擎**：遍历指令数组，对每条指令执行对应的 MongoDB 操作（`findOneAndUpdate`、`create`、`deleteOne`），操作失败不中断后续

### 4.5 情感陪伴智能体实现（约 1,200 字，**论文重点技术贡献**）

**写作要点**：

这是论文的核心创新点之一。需要详细阐述如何通过 Prompt 工程实现深度情感陪伴，替代开题报告中计划的模型微调方案。

1. **Prompt 架构设计**（四层结构）：

   **第一层 — 角色设定**：定义 AI 助手的核心特质（善于倾听、精准感知、温暖但不过度、动态调整、自然不模板化）

   **第二层 — 用户画像注入**（`buildProfileContext` 函数）：
   - 新老用户区分（interactionCount 阈值：≤3 新用户热情引导、>50 老用户简洁高效）
   - 近期情绪趋势分析（最近 5 次交互中负面情绪 ≥3 次则标记"需要额外关心"）
   - 任务完成率反馈（≥80% 给予肯定、<50% 温和鼓励不施压）
   - 工作风格偏好适配（intense/balanced/relaxed）

   **第三层 — 八类情绪响应策略**：为每类情绪设计差异化的回复策略框架
   - anxious：验证焦虑合理性 → 拆分小步骤 → 禁止"别紧张"式否定
   - sad：陪伴优先 → 不急于建议 → 禁止"振作起来"式强迫乐观
   - stressed：共情 → 提出简化方案 → "我来帮你理一理"
   - tired：关心身体 → 建议推迟非紧急 → 弱化提醒
   - angry：认同合理性 → 不评判 → 等待表达完毕
   - confused：梳理思路 → 结构化选项 → 不代替决策
   - happy：积极回应 → 借势推动 → 活泼轻松
   - neutral：高效沟通 → 简洁明了

   **第四层 — 回复约束规则**：长度 100-200 字、先情绪后事务、禁止模板化、自然融入操作结果确认

2. **上下文整合机制**：
   - 对话历史：加载最近 10 条记录，标注当前轮次
   - 操作结果：将 taskAgent/scheduleAgent 的执行结果（如"已创建1个任务：开会"）自然嵌入回复
   - 情绪上下文：将 routerAgent 检测到的情绪类型、强度、触发因素传入

3. **降级策略**：LLM 调用失败时使用预定义的 8 类情绪降级回复，保证用户始终得到响应

4. **与微调方案的对比论证**：
   - Prompt 工程方案：无需训练数据、无需 GPU 资源、迭代速度快、BM-6 评分 4.31/5
   - 微调方案：需要大量标注数据、训练成本高、模型版本管理复杂
   - 结论：对于本场景，Prompt 工程已达到充分的情感回复质量
